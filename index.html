    I'm a software developer in Cambridge, Massachusetts. You can reach me at jimkang@fastmail.com.

    I like matching problems to ideas. Then, I like to make the ideas real and see if they live or die.

    I have been doing this for years by writing code to create apps, APIs, and tools. (I have been a software engineer for nearly two decades.)

    This is an overview of my work. If you just want to go straight to my publicly available code, here are my Github repos and my NPM packages.

    • Jobs
      • Web and mobile era
      • This part of my career started around 2008 and continues today.
        • Spotify
        • I currently work at Spotify. In 2016-2018, I worked on a team that built music recommenders for users (e.g. the This Is playlists — greatest hits playlists for artists that stay fresh by updating daily based on listening patterns) — and Time Capsule). Now, I work on a team that is building voice experiences.

          Some things I've worked on at Spotify:

          • An image generator that composes thousands of playlist covers each day based on playlist contents
          • Internal apps (web apps, Chrome extensions, Electron apps) for evaluating and adjusting algorithmically-generated content
          • Interaction and content prototypes
          • Voice experience prototypes built in Alexa Skills and the browser (using the Web Speech API and DialogFlow)
          • External promotional web sites
          • Gathering data from users via surveys

          My work involves full-stack web engineering, navigating a unique internal infrastructure. It also involves investigating both user needs and technical possibilities.

          There is quite a bit of building things purely to see how users will react (in in-person user tests and A/B tests). I've learned much about the value and costs of gathering information before you commit to something.

        • PayPal
        • I worked at PayPal on the Shop, a system built in Node.
          The Shop site served coupons from various PayPal partners to hundreds of thousands of people per day. The front end was a single-page app built in AngularJS. The back end was a cluster of NodeJS servers that talked to PayPal services to manage the coupon and user information.
          We used TDD, pair programming, and GitFlow heavily. We prioritized the mobile web experience, which I've taken to heart ever since. And, perhaps less effectively, Scrum.
        • NPR
        • I worked at NPR on a station management app and API. It was called Composer, and it was written in Node and Backbone. Over 200 NPR member stations used it to add program schedules to their web sites and keep track of what they played so they could pay for it.

          I added this app's first automated tests (in order to safely do a big refactor to add OAuth to the API) and learned quite a bit about TDD as a result. Our team used a Kanban process and an unusually high level of design-development integration.

        • Modo Labs

        • The company built mobile apps for big institutions, mostly universities. It also wrapped up customers' existing data sources to make them easily mobile-consumable.

          I worked on various universities' iOS apps here.
        • Voter Activation Network

        • Their business was centered around a massive data warehousing web application.

          I worked on their ASP .NET and iOS apps.
        • Obama for America
        • I spent a month as a "Data Fellow" during the 2008 presidential campaign.

          Indiana went blue, which it had not done since 1964.

          We automated manual tasks and wrote programs to look through voter contact history to find good targets for calls. I met fantastic people, and I saw how the relatively simple application of technology can make a big impact when applied to the right context.

      • Windows era
      • This era lasted from about right after college (1999) to 2007.
        • GN Resound

        • GN Resound is a hearing aid company. Hearing aids are small digital sound processors. Audiologists can calibrate and gather data from them by connecting them to a computer.

          I worked on their hearing device adjustment software, which at the time, was a C++ COM/ATL-based Windows desktop application.

        • InstallShield

        • This is where I learned to write code that worked in real-world situations when run by thousands (millions in a few cases) of real users.

          I worked on IDEs that let developers author "setups," programs that installed software onto computers. The IDEs were Windows applications written in C++.

      • annoy-node is Node bindings for Annoy, a popular Approximate Nearest Neighbors implementation in C++.

        There are a lot of machine learning models that figure out what is similar to what else. The way they express what they've figured out is in n-dimensional vectors representing things like words or songs.

        This Node module lets you use — in JavaScript — what those ML models have produced.

    • What I'm looking for
    • I want to build products that
      • Help people understand ideas and situations.
      • Get people to imagine possibilities.
      • Put power in the hands of users instead of taking it from them. (e.g. Does not make decisions on their behalf that may not be in their interest.)
      • Get people to think instead of just consume. I don't think this has to be onerous or burdensome.
    • I want to build technology that
      • Is honest and predictable - does what it says it will do.
      • Does as little as possible.
    • I am interested in
      • Art
      • Visual explanations
      • Procedural generation
      • Design
    • To do my best work, I need
      • Full understanding of a problem, so it can be addressed efficently, with a wide set of possible tactics (e.g. NOT "Use x to do y" since x may not actually be the best way to accomplish y.)
      • As few arbitrary organizational restrictions as possible. (e.g. "No, only designers do that" or "That is up to product."
      • A culture that values taking the time to understand problems and the space that they're working in instead of wallpapering over it.
        • This means sometimes writing your own code instead of pulling in a framework or module that you think does what you want (along with several other things).
        • It might mean being slower in the first iteration but faster in subsequent iterations because you understand the what you're working with.
